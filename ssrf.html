# SSRF Injection

SSRF (Server-Side Request Forgery)

SSRF is a vulnerability where an attacker tricks a server into sending requests to other locations.

Flow:

Attacker → Vulnerable Server → Internal Service

The attacker abuses the server like a proxy.

1. BASIC SSRF

In Basic SSRF, the response is visible to the attacker.

Example

POST /stockApi
stockApi=http://localhost/admin

The server sends:

GET /admin HTTP/1.1
Host: localhost

What Happens?

The server accesses its own localhost. If response is shown, attacker may see: 

- admin panel
- internal data
- private APIs

How to Identify Basic SSRF

Look for features that fetch URLs:

- image fetcher
- stock checker
- URL preview
- webhook

Test with:

http://127.0.0.1

If response changes → SSRF may exist.

2. BLIND SSRF

In Blind SSRF, request is sent but response is NOT visible. So the attacker needs another way to confirm the request. How Blind SSRF Works

Payload

http://abc123.oastify.com

http://abc123.interact.sh

If the server visits that domain, you receive a DNS/HTTP interaction. That confirms SSRF..

Detection Tools

- Burp Collaborator
- Interactsh
- DNS logging services

Flow

Attacker
↓
Vulnerable Website
↓
Collaborator Domain

If interaction appears → Blind SSRF confirmed.

3. METADATA ACCESS CONCEPT

Cloud servers often have internal IP: 

169.254.169.254

This is called the metadata service.

AWS Example

http://169.254.169.254/latest/meta-data

Sometimes it contains:

- tokens
- instance information
- IAM credentials
- network details

latest
└── meta-data
└── iam
└── security-credentials
└── ROLE_NAME

Why Dangerous?

Normally external users cannot access it. But with SSRF: Attacker → Server → Metadata Service The server accesses it internally.

## Important Concept - not internet accessible SSRF bypasses that limitation..

4. URL PARSING BYPASS CONCEPT

Sometimes developers block: 

127.0.0.1
localhost
0.0.0.0
[::1]

But filters are not always perfect. Attackers try different URL formats to bypass weak validation.

## Important Idea The application may: - validate the URL one way - but request the URL another way This mismatch can create bypasses. --- ## Main Goal Trick the filter into thinking: "This URL is safe" while the server actually connects to an internal target.

COMMON SSRF TARGETS

- localhost
- internal APIs
- admin panels
- cloud metadata
- private network services

COMMON SSRF PARAMETERS

url=
link=
api=
feed=
callback=
redirect=
image=

SSRF vs XXE

XXE can sometimes cause SSRF. 

<!ENTITY xxe SYSTEM "http://internal-api/admin">

So:

XXE → SSRF possible

SIMPLE ANALOGY

You cannot enter a restricted room. But you tell a security guard: "Please go inside and check it." The guard can access places you cannot.

In SSRF: Server = Security Guard
